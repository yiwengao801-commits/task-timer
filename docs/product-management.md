# TaskTimer 产品文档

## 1. 产品定位

TaskTimer 是一个面向桌面办公场景的 **任务计时 + 日程记录 + 效率工具**。

它的核心价值不是复杂项目管理，而是让用户用最低的操作成本完成两件事：
- 记录“我今天做了什么、花了多久”
- 在工作时快速打开常用网页

产品形态以 **Electron 桌面应用** 为主，兼顾“主页面完整管理”和“桌面悬浮组件高频操作”两种使用场景。

---

## 2. 目标用户

- 需要记录工作时间的自由职业者、远程工作者
- 希望快速追踪任务耗时的开发者、设计师
- 希望在桌面保留常用网页入口的办公用户
- 需要“最小化打扰”的专注型工作者

用户共同点：日常在电脑前工作，希望工具轻量、不抢注意力、随手可用。

---

## 3. 核心设计理念

### 3.1 设计原则

- **轻量优先**：避免复杂配置，打开即可计时
- **桌面原生感**：悬浮窗、置顶、最小化，像系统工具一样存在
- **数据可控**：所有记录保存在本地，不依赖云端
- **低打扰健康提醒**：定时提醒喝水、走动、护眼、伸展，文案风格轻松幽默

### 3.2 信息架构

产品分为三层：

**第一层：主页面（完整管理）**
- 计时器主控
- 今日日程列表
- 日历视图
- 多日期选择 + 报告导出
- 快捷入口管理

**第二层：桌面悬浮组件（高频操作）**
- 计时器悬浮窗：开始/暂停/结束，最小化、置顶、关闭
- 快捷入口悬浮窗：4 宫格网页快捷方式，点击即开

**第三层：系统托盘（后台常驻）**
- 应用最小化到托盘后不退出
- 托盘菜单可重新呼出主窗口和悬浮窗

## 4. 功能迭代历程

### 4.1 第一阶段：核心计时 + 日程记录

**需求来源**
用户最初需要一个“能输入任务名称、开始计时、自动记录时间段”的工具，并且希望以日历形式查看历史记录。

**实现内容**
- React + Vite 主页面
- 计时器组件：输入任务名、开始/暂停/结束
- 日程组件：按天展示任务列表，支持编辑和删除
- 日历组件：月视图，标记有任务的日期
- 数据存储：localStorage，按日期分组

**设计决策**
- 优先做本地版本，不先做云端同步
- 采用“日历 + 日程列表”双视图，兼顾概览和详情

---

### 4.2 第二阶段：报告导出 + 多选模式

**需求来源**
用户希望在日历中勾选多个日期，然后生成指定时间段的工作报告。

**实现内容**
- 日历多选模式：勾选日期范围
- 报告生成弹窗：汇总所选日期的任务、时长
- 将“多选模式”按钮改为“选择导出”，降低理解成本

**设计决策**
- 多选状态用顶部状态栏展示，不占用日历空间
- 报告以弹窗形式呈现，不跳转新页面

---

### 4.3 第三阶段：Electron 桌面化

**需求来源**
用户希望工具作为桌面应用运行，具备悬浮窗、置顶、最小化等桌面应用特性。

**实现内容**
- 引入 Electron，创建主窗口和悬浮窗
- 计时器桌面组件：无边框、透明背景、始终置顶
- 系统托盘集成：关闭主窗口时最小化到托盘
- IPC 通信：桌面组件与主页面之间的任务数据同步

**设计决策**
- 桌面组件采用独立 HTML 文件，与 React 主页面解耦
- 关闭按钮实际为“隐藏”，通过托盘菜单可重新显示
- 最小化状态压缩为仅显示标题栏，减少占用空间

---

### 4.4 第四阶段：健康提醒系统

**需求来源**
用户希望工具在长时间使用后主动提醒休息，避免久坐伤身。

**实现内容**
- 四类提醒：喝水、走动、护眼、伸展
- 随机时间触发：40–60 分钟随机间隔
- 随机文案：每类提醒准备多条幽默文案，随机展示
- 提醒窗口：无边框弹窗，支持关闭和“10 分钟后再次提醒”

**设计决策**
- 文案风格偏吐槽、幽默，避免说教感
- 提醒窗口始终置顶，确保用户能看到
- 不打断当前操作，用户可随时关闭

---

### 4.5 第五阶段：快捷网页入口

**需求来源**
用户希望在桌面快速打开常用网页，减少切换浏览器的成本。

**实现内容**
- 主页面快捷入口管理弹窗：添加/编辑/删除最多 4 个入口
- 图标和颜色自定义：预设图标 + 纯色调色板
- 快捷入口桌面悬浮窗：4 宫格布局，点击直接打开网页
- Electron 集成：通过系统默认浏览器打开链接

**设计决策**
- 桌面悬浮窗与计时器悬浮窗独立，可分别显示/隐藏
- 快捷入口数据统一存储在 Electron 主进程，避免 localStorage 分片问题
- 旧数据自动迁移：将渐变背景 `bg` 转为纯色 `color`

---

### 4.6 第六阶段：UI 精细化

**需求来源**
用户反馈桌面组件尺寸偏大，希望压缩高度、保持宽度不变。

**实现内容**
- 缩小计时器显示字体：42px → 24px
- 压缩组件内边距和元素间距
- 调整按钮和输入框尺寸
- 优化最小化状态下的视觉呈现

**设计决策**
- 宽度固定 280px，高度尽量压缩
- 最小化状态保持 48px 高度，仅显示标题栏

## 5. 技术架构

### 5.1 技术栈

- **前端框架**：React 18
- **构建工具**：Vite
- **桌面框架**：Electron 28
- **数据存储**：localStorage（早期）+ Electron 主进程 JSON 文件（统一存储方案）
- **CI/CD**：GitHub Actions

### 5.2 架构分层

```
┌─────────────────────────────────────────┐
│            主页面（React）                │
│  - 计时器、日程、日历、报告、快捷入口管理  │
├─────────────────────────────────────────┤
│         Electron 主进程                  │
│  - 窗口管理、IPC、托盘、提醒、存储        │
├─────────────────────────────────────────┤
│         桌面悬浮组件（独立 HTML）         │
│  - 计时器悬浮窗、快捷入口悬浮窗          │
└─────────────────────────────────────────┘
```

### 5.3 数据流

- **任务数据**：桌面组件 → IPC → 主进程 → 主窗口 → 保存到统一存储
- **快捷入口数据**：主页面管理 → IPC → 主进程 JSON → 桌面悬浮窗读取
- **健康提醒**：主进程定时器 → 提醒窗口

## 6. 发布流程

### 6.1 版本管理

- 版本号维护在 `package.json` 中
- 当前版本：`1.0.0`
- 通过 Git tag 触发正式发布，GitHub Actions 自动打包

### 6.2 CI/CD 流程

项目已配置 GitHub Actions 自动发布工作流（`.github/workflows/release.yml`）：

1. **触发方式**
   - 推送 `v*` 标签：触发正式 Release
   - 手动 `workflow_dispatch`：触发预览构建

2. **构建矩阵**
   - Windows (`windows-latest`) → `.exe`
   - macOS (`macos-latest`) → `.dmg`
   - Linux (`ubuntu-latest`) → `.AppImage`

3. **产物上传**
   - 各平台构建产物上传为 GitHub Actions Artifacts
   - 自动创建 GitHub Release，附上所有安装包

4. **用户获取方式**
   - 访问 GitHub Releases 页面
   - 下载对应平台的安装包
   - 双击安装/运行，无需 Node.js 或其他依赖

### 6.3 本地打包（备用）

如果需要在本地打特定平台包：

```bash
# 通用构建
npm run build

# macOS 本地打包
npm run electron:build

# 通过 Docker 在 macOS 上打 Windows 包
docker run --rm -it --platform=linux/amd64 \
  -v "$PWD":/project \
  -w /project \
  electronuserland/builder:wine \
  bash -lc "npm ci || npm install && npm run build && npx electron-builder --win --x64"
```

## 7. 已知问题与后续规划

### 7.1 已知问题

- **开发模式与正式模式数据分片**：localStorage 按 origin 隔离，开发模式（localhost）和正式模式（file://）的任务记录不共享
- **快捷入口旧数据迁移**：早期版本使用渐变背景 `bg` 字段，新版本改为纯色 `color`，已做自动迁移但需验证

### 7.2 后续规划

- **统一存储迁移**：将任务记录也从 localStorage 迁移到 Electron 主进程统一存储，彻底解决分片问题
- **数据导入/导出**：支持 JSON 格式的任务数据备份和恢复
- **更多快捷入口**：支持超过 4 个入口，或按分组管理
- **自定义提醒间隔**：允许用户设置提醒频率
- **系统托盘快捷操作**：托盘图标右键直接开始/暂停计时

## 8. 设计决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 桌面组件独立 HTML | 是 | 与 React 主页面解耦，加载更快，减少依赖 |
| 数据统一存储在主进程 | 是 | 解决多窗口/多 origin 数据分片问题 |
| 健康提醒随机文案 | 是 | 降低重复感，增加使用愉悦度 |
| 快捷入口上限 4 个 | 是 | 保持桌面组件紧凑，避免信息过载 |
| 发布方式 GitHub Actions | 是 | 自动化构建，无需本地打包环境 |
| 托盘菜单呼出悬浮窗 | 是 | 用户关闭悬浮窗后仍可快速找回 |
